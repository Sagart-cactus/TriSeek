# TriSeek v0.3.3

TriSeek v0.3.3 expands Codex hook coverage and adds a Docker-isolated real execution harness for release validation.

## Highlights

- Added Codex `PreToolUse` hooks alongside `PostToolUse` and `SessionStart`.
- Added Codex hook matchers for Bash reads and MCP file-read tools such as `mcp__filesystem__read_file`.
- Taught the Memo hook shim to classify MCP file-read tool names as read events.
- Updated Codex, OpenCode, install, MCP, and Memo documentation to describe the current hook surfaces accurately.
- Added a Docker real harness that builds release binaries, runs CLI checks, starts an isolated daemon, drives MCP stdio tools, verifies search reuse invalidation, and smoke-tests install/uninstall config generation for Claude Code, Codex, OpenCode, and Pi.
- Added optional large-repo smoke coverage via `TRISEEK_LARGE_REPO`.

macOS and Linux:

```sh

```

Windows PowerShell:

```powershell
powershell -ExecutionPolicy Bypass -c "irm https://raw.githubusercontent.com/Sagart-cactus/TriSeek/main/scripts/install.ps1 | iex"
```

## Notes

- Codex passive Memo coverage depends on the installed Codex hook surface. Bash reads and MCP file-read tools can be observed through hooks; non-hooked reads should still use `memo_check`.
- The Docker harness isolates `HOME`, `USERPROFILE`, `XDG_CONFIG_HOME`, `TRISEEK_HOME`, and daemon loopback state so release validation does not touch the user's real daemon or client configs.
