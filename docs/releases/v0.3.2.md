# TriSeek v0.3.2

TriSeek v0.3.2 adds context-aware search result reuse for MCP search tools. When the same search runs again in the same harness session and no relevant files changed, TriSeek now tells the model to reuse the earlier result already in context instead of paying for the full output again.

## Highlights

- Added MCP-local search memo tracking with stable `search_id`s for repeated `find_files`, `search_content`, and `search_path_and_content` calls.
- Added daemon-side freshness checks backed by watcher generation tracking, changed-path journals, and context epoch invalidation.
- Invalidated duplicate-search reuse when matching files or scoped files change, when journal coverage is exhausted, or when the harness session context is compacted.
- Extended harness integration so Claude, Pi, and OpenCode can explicitly invalidate search reuse on session start and compaction boundaries.
- Added a separate search-reuse validation runner alongside the existing Memo replay tooling and verified the new search savings independently of the older memo-only study.

## Notes

- Duplicate-search reuse is conservative by design: if TriSeek cannot prove freshness, it reruns the search and returns full results.
- Codex currently gets the core duplicate-search optimization, but it still lacks a verified pre-compact hook surface comparable to Claude, Pi, and OpenCode.
