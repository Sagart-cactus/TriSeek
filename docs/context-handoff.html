Switch agents without
losing your place

Context Handoff snapshots your current session — goal, open files, recent searches, git state, and pinned snippets — and prints a one-line resume command for the receiving harness. No cold start. No re-discovery. Agent switching made explicit.

// 01 — overview

Portable session state

A handoff snapshot is a self-contained envelope that captures everything the receiving agent needs to continue work where you left off — without re-reading files or re-running searches from scratch.

→

What a snapshot contains

goal · The active task description from the originating session
working_set · Files opened or read during the session, with read timestamps
action_log · Chronological log of searches, file reads, and session events
pinned_snippets · Code excerpts explicitly marked for carry-over
git_state · Commit hash, branch, and list of dirty files at snapshot time
briefing · Markdown summary: goal, hypothesis, established facts, open questions
◕

What the receiving agent gets

When the target harness runs resume, TriSeek loads the snapshot and injects the briefing directly into the agent's context. The agent reads:

  • The goal and current hypothesis
  • Which files have already been read (memo avoids re-reading unchanged files)
  • Which searches have already been run (search reuse avoids duplicate calls)
  • Any pinned code snippets from the originating session
  • The git commit and dirty-file list so branch state is unambiguous

The agent can pick up the next planned action immediately.

// 02 — claude → codex

Handing off from Claude

Work in Claude Code, create a snapshot, then paste the resume command into Codex. The handoff block tells you exactly what to paste.

1

Work in Claude

Open a session, search files, read code, and make progress on a task. TriSeek tracks the files you read and the searches you run automatically through the session.

2

Run the handoff command

When you are ready to switch, run the handoff slash command in Claude Code:

Claude Code
/triseek handoff codex

TriSeek creates a snapshot of the session and prints a handoff block:

TriSeek handoff ready
 
From: Claude
To: Codex
Session: my-session
Snapshot: snap_1746300000_my-session
Brief: ~/.triseek/daemon/snapshots/snap_1746300000_my-session/briefing.md
 
In Codex, paste:
$triseek resume snap_1746300000_my-session
3

Paste the resume command in Codex

Open Codex and paste the $triseek resume command shown in the handoff block. Codex loads the snapshot and continues from where Claude left off.

Codex
$triseek resume snap_1746300000_my-session
// 03 — codex → claude

Handing off from Codex

The reverse flow works identically. Codex uses $triseek handoff and Claude resumes with /triseek resume.

1

Work in Codex

Run searches, read files, and make progress. TriSeek records the session through its MCP tools as usual.

2

Run the handoff command

When ready to switch to Claude, run:

Codex
$triseek handoff claude

The printed block shows what to paste in Claude Code:

TriSeek handoff ready
 
From: Codex
To: Claude
Session: my-session
Snapshot: snap_1746300060_my-session
Brief: ~/.triseek/daemon/snapshots/snap_1746300060_my-session/briefing.md
 
In Claude, paste:
/triseek resume snap_1746300060_my-session
3

Paste the resume command in Claude

Open Claude Code and paste the /triseek resume command. Claude loads the snapshot and inherits the session state from Codex.

Claude Code
/triseek resume snap_1746300060_my-session
↕

Command summary

The prefix differs by harness — / for Claude Code slash commands, $ for Codex shell commands. The handoff block always prints the correct command for the target, so you do not need to remember the difference.

Claude Code

initiate
/triseek handoff codex
resume
/triseek resume <snapshot_id>

Codex

initiate
$triseek handoff claude
resume
$triseek resume <snapshot_id>
// 04 — why it helps

No more cold starts

Every agent session that starts without context wastes tokens re-reading files and re-running searches already done in the previous session. Context Handoff eliminates that overhead.

⛌

Without handoff

  • New session starts with no awareness of the previous session's work
  • Agent re-reads files already read in the prior session
  • Agent re-runs searches already completed with known results
  • Progress on the goal must be re-explained from scratch
  • Agent may take a different investigative path and reach different conclusions
✓

With handoff

  • Goal preserved — the receiving agent knows what was being worked on
  • Files preserved — memo reuse skips re-reading unchanged files already in the working set
  • Searches preserved — search reuse avoids duplicate MCP calls for already-known results
  • Facts preserved — established facts and ruled-out hypotheses carry over in the briefing
  • Git state preserved — the receiving agent knows which commit and dirty files the snapshot was taken on
ⓘ

Explicit switching as a feature

Context Handoff makes agent transitions explicit and inspectable. The snapshot is a readable file on disk — you can open briefing.md in any editor to see exactly what will be passed to the receiving agent. There are no hidden state transfers or opaque serialized contexts.

// 05 — before you start

Check MCP registration

Both harnesses must have the TriSeek MCP server registered before a handoff will work. Run triseek doctor to verify.

◆

triseek doctor

The doctor command checks that the TriSeek daemon is running and that the MCP server is reachable. Run it from the repo root before starting a handoff demo or workflow.

shell
triseek doctor

A healthy output shows the daemon port, index status, and confirms the MCP server socket is responding. Fix any flagged issues before testing handoff.

⚠

What to check

  • Daemon running — triseek doctor should report a live PID and port
  • MCP registered in Claude — check .mcp.json or run claude mcp list and look for triseek
  • MCP registered in Codex — check ~/.codex/config.toml for the TriSeek MCP entry
  • Repo indexed — run triseek index-status to confirm the trigram index is up to date

See the install guide and MCP reference for registration instructions.

Context Handoff release

Context Handoff shipped in TriSeek v0.5.0. It requires the TriSeek daemon (introduced in v0.3.0) and the session tools (session_open, session_snapshot, session_handoff, session_resume) available via MCP.

The briefing format is a plain Markdown file — no proprietary encoding. You can read, edit, or diff snapshots with any text tool.
