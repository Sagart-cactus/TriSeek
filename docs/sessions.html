Session Lifecycle

A work session is a goal plus a series of TriSeek calls plus zero or more snapshots. Lifecycle: Open → work and snapshots → Closed as resolved or abandoned.

// model

The session model

session_open records a goal and repo root. Search, memo, frecency, and context-pack calls can accrue to the action log. session_snapshot captures that state for later resume.

State diagram

Open
  | work: search, memo, context_pack
  | snapshot: zero or more checkpoints
  v
Closed
  | resolved | abandoned
// mcp tools

MCP tools

ToolUsePattern
session_openStart tracking work.Call once when the goal is known.
session_statusInspect current state.Use before snapshotting or closing.
session_listFind existing sessions.Useful when two harnesses share the daemon.
session_closeMark resolved or abandoned.Close only when the handoff or work is done.
session_snapshotCapture working set and action log.Use before switching harnesses.
session_handoffSnapshot and close in MCP.Use CLI handoff when target-aware files are needed.
session_resumePrime daemon state and return markdown payload.Use at the start of the receiving harness.
// cli

CLI walkthrough

triseek snapshot create --session session_123 --source-harness claude_code .
triseek snapshot list
triseek snapshot show <snapshot_id>
triseek brief <snapshot_id> --mode no-inference
triseek pack export <snapshot_id> --output handoff.tcp
triseek resume <snapshot_id> --write-to AGENTS.md

// multiple harnesses

Multiple harnesses, one session

Claude Code and Codex can both refer to the same session id because the state lives in the long-running TriSeek daemon, not in either harness. That shared daemon is what lets resumed sessions inherit memo, frecency, and warmed search reuse.
